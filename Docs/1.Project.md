## Descrição

O Projeto Bigtech é uma plataforma SaaS **multi-usuário com separação lógica por cliente (tenant)**, orientada a plugins, destinada à consulta de dados de crédito, cadastrais, veiculares e imobiliários.

⚠️ **Decisão estratégica imutável:**
A plataforma opera como **single-tenant em nível de infraestrutura** e **multi-usuário / multi-organização em nível de aplicação**.
Não existe isolamento por containers, instâncias, ambientes ou projetos por tenant. Todo isolamento ocorre **exclusivamente via autenticação, autorização e modelagem de dados**.

A plataforma se posiciona como um birô técnico de informação, orquestrando, normalizando, controlando acesso e monetizando consultas de múltiplas fontes de dados, **sem deter os dados propriamente ditos**.

A arquitetura é dividida em um **CORE mínimo, estável e agnóstico**, com funcionalidades variáveis implementadas via plugins, permitindo extensibilidade, governança e manutenção simplificada.

---

## Objetivo

* Criar um MVP web-app de consultas para o mercado varejista.
* Suportar múltiplos clientes (**tenants lógicos**) e múltiplos usuários **na mesma infraestrutura compartilhada**.
* Garantir isolamento **exclusivamente lógico** de dados, permissões e configurações.
* Permitir instalação, ativação e desativação de funcionalidades via sistema de plugins.
* Preparar a base para uma API pública voltada ao mercado atacadista (fora do MVP).
* Minimizar código hardcoded no CORE, focando em estabilidade e agnosticismo.
* Facilitar a evolução do produto através de plugins versionáveis, substituíveis e comercializáveis.

**Observação operacional:**
Todas as ações administrativas que modificam o estado de plugins (instalar, habilitar, desabilitar, configurar, remover) **devem produzir um `auditId`** retornado pela API do CORE. As UIs administrativas devem exibir esse `auditId` ao final da operação para rastreabilidade e investigação.

---

## O que não é

O Projeto Bigtech **não** é:

* Uma plataforma com isolamento por containers ou instâncias por cliente
* Um sistema multi-tenant em nível de infraestrutura
* Uma detentora de dados primários ou proprietários
* Um monólito com regras de negócio acopladas ao CORE
* Uma API pública aberta sem controle de plugins e billing

O CORE **não** realiza consultas de dados, integrações externas específicas ou pagamentos diretos.
Funcionalidades variáveis **nunca** residem no CORE.

## Como funciona na prática

Na prática, a plataforma opera com três frontends principais:

* **App do usuário final:** `app.bigtech.com.br`
* **Administração:** `admin.bigtech.com.br`
* **API pública atacadista:** implementação futura (fora do MVP)

O CORE gerencia:

* Autenticação
* Autorização
* Separação lógica de tenants
* Permissões de usuários
* Carregamento dinâmico de plugins
* Billing genérico
* Auditoria

Plugins são categorizados em:

* **Fontes de dados** (ex.: `plugin-infosimples`)
* **Consultas** (ex.: `plugin-consulta`)
* **Pagamentos** (ex.: `plugin-pagamento-asaas`)
* **Mercados** (ex.: `plugin-varejo`, `plugin-atacado`)
* **Funcionais**

Usuários fazem login, compram créditos, executam consultas via plugins ativos, com fallback configurável.
Administradores gerenciam **clientes (tenants lógicos)**, usuários, plugins e configurações.

A comunicação entre plugins ocorre via CORE e Event Bus, garantindo isolamento lógico, extensibilidade e auditabilidade.

---

## Gestão de Plugins

A gestão de plugins é uma funcionalidade central do admin:

* Plugins podem ser instalados globalmente
* Habilitados/desabilitados por tenant lógico
* Configurados (`apiKey`, `fallbackSources`, etc.)
* Atualizados de forma controlada

O CORE expõe um `pluginLoader` responsável por:

* Ativação dinâmica por tenant lógico
* Validação de dependências entre plugins
* Emissão de eventos de auditoria (`Audit`)

Todas as decisões seguem o princípio de **CORE mínimo**, mantendo regras variáveis exclusivamente nos plugins.

---

## Pilar Central do Projeto Bigtech

> **“CORE mínimo, estável e agnóstico. Tudo que muda é plugin.”**

Esse pilar é inegociável e governa toda a arquitetura, documentação e implementação do sistema.

---

## Modelo de Negócio

O modelo de negócio é SaaS **multi-usuário com tenants lógicos**, com monetização baseada em consumo de consultas.

* Clientes (tenants) compartilham a mesma infraestrutura
* Dados e configurações são isolados logicamente
* Cobrança por uso ou planos pré-pagos
* Plugins podem ser comercializados separadamente
* Upsell e customização via ativação de plugins

Não há dependência de provedores fixos; integrações financeiras ocorrem via plugins.

---

## Glossário

* **CORE**: Núcleo mínimo e estável responsável por autenticação, autorização, tenants lógicos, plugins e auditoria.
* **Plugin**: Módulo extensível que implementa funcionalidades variáveis.
* **Tenant (lógico)**: Cliente da plataforma, isolado por dados, permissões e configuração — **não por infraestrutura**.
* **Multi-usuário**: Múltiplos usuários operando no mesmo sistema, com controle de acesso.
* **SaaS**: Software como serviço.
* **Event Bus**: Comunicação assíncrona entre CORE e plugins.
* **Audit**: Registro imutável de ações administrativas.
* **Billing Engine**: Mecanismo genérico de monetização.
