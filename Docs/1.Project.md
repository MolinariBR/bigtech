## Descrição

O Projeto Bigtech é uma plataforma SaaS multi-tenant, orientada a plugins, destinada à consulta de dados de crédito, cadastrais, veiculares e imobiliários. A plataforma se posiciona como um birô técnico de informação, orquestrando, normalizando, controlando acesso e monetizando consultas de múltiplas fontes de dados, sem deter os dados propriamente ditos. A arquitetura é dividida em um CORE mínimo e estável, com funcionalidades variáveis implementadas via plugins, permitindo extensibilidade e manutenção simplificada.

## Objetivo

- Criar um MVP web-app de consultas para o mercado varejista.
- Suportar múltiplos tenants (clientes) com isolamento (docker) e personalização.
- Permitir instalação, ativação e desativação de funcionalidades via sistema de plugins.
- Preparar a base para uma API pública voltada ao mercado atacadista.
- Minimizar código hardcoded no core, focando em estabilidade e agnosticismo.
- Facilitar a evolução do produto através de plugins versionáveis, substituíveis e comercializáveis.

Observação operacional: todas as ações administrativas que modificam o estado de plugins (instalar, habilitar, desabilitar, configurar, remover) devem produzir um `auditId` retornado pela API do CORE; as UIs administrativas devem exibir esse `auditId` ao final da operação para rastreabilidade e investigação.

## O que não é

O Projeto Bigtech não é uma detentora de dados primários; não armazena ou processa dados proprietários diretamente. O CORE não contém regras de negócio específicas, não realiza consultas de dados, integrações externas ou pagamentos específicos. Não é uma plataforma monolítica; evita hardcoding de funcionalidades variáveis. Não expõe API pública sem o plugin adequado ativado, e não permite acesso irrestrito aos dados.

## Como funciona na pratica

Na prática, a plataforma opera com três frontends principais: um para usuários finais (app.bigtech.com.br), um para administradores (admin.bigtech.com.br) e uma API pública opcional para o mercado atacadista que não deve entrar no MVP, será uma implementação futura. O CORE gerencia autenticação, multi-tenancy, permissões, carregamento de plugins e billing genérico. Plugins são categorizados em fontes de dados (ex.: plugin-infosimples), consultas (ex.: plugin-consulta), pagamentos (ex.: plugin-pagamento-asaas), mercado (ex.: plugin-varejo/plugin-atacado) e funcionais. Usuários fazem login, compram créditos, executam consultas via plugins ativos, com fallback configurável. Administradores gerenciam tenants, plugins e configurações. A comunicação entre plugins ocorre via CORE e Event Bus, garantindo isolamento e extensibilidade.

Adicionalmente, a gestão de plugins é uma funcionalidade central do admin: plugins podem ser instalados, habilitados/desabilitados por `tenant`, configurados (ex.: `apiKey`, `fallbackSources`) e atualizados. O CORE expõe um `pluginLoader` responsável por ativação dinâmica por tenant, validação de dependências entre plugins e emissão de eventos de auditoria (`Audit`) para todas as mudanças de plugins (instalação, habilitação, configuração, remoção). As decisões de ativação são tratadas com cautela para preservar o princípio "CORE mínimo" e manter regras variáveis nos próprios plugins.

## Pilar Central do Projeto Bigtech

O pilar central é: "CORE mínimo, estável e agnóstico. Tudo que muda é plugin." Esse princípio garante que o núcleo da plataforma permaneça imutável e focado em funcionalidades transversais, enquanto inovações e variações são encapsuladas em plugins independentes, facilitando manutenção, escalabilidade e comercialização.

## Modelo de negocio

O modelo de negócio é SaaS multi-tenant, com monetização baseada em consumo de consultas. Clientes (tenants) adquirem créditos ou planos para executar consultas via plugins ativos. Para o mercado varejista, acesso via web-app com cobrança por uso. Para o atacadista, via API pública, para usarem em suas proprias soluções, mas sempre pagamento com modelo pre-pago. Plugins podem ser comercializados separadamente, permitindo customização e upsell. A plataforma controla acesso, audita logs e integra pagamentos via plugins específicos, sem dependência de provedores fixos.

## Glossãrio

- **CORE**: Núcleo mínimo e estável da plataforma, responsável por funcionalidades transversais como autenticação, multi-tenancy e plugin loader.
- **Plugin**: Módulo extensível que implementa funcionalidades variáveis, como fontes de dados, consultas, pagamentos ou mercados.
- **Multi-tenant**: Arquitetura que suporta múltiplos clientes isolados na mesma instância da plataforma.
- **SaaS**: Software as a Service, modelo de entrega de software via nuvem.
- **MVP**: Minimum Viable Product, versão inicial mínima do produto para validação.
- **Event Bus**: Sistema de comunicação assíncrona entre componentes da plataforma.
- **Fallback**: Mecanismo de contingência para usar fontes alternativas em caso de falha.
- **Billing Engine**: Componente genérico para cobrança e monetização.
- **Tenant**: Cliente individual da plataforma, com isolamento de dados e configurações.
